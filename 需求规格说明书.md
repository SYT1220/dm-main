# 需求规格说明书

## 1. 引言

### 1.1 编写目的

本文档旨在明确宿舍管理系统的功能需求、性能需求、接口需求等，为系统设计、开发、测试提供依据，同时作为项目验收的标准。本文档面向项目管理人员、系统设计人员、开发人员、测试人员以及最终用户。

### 1.2 项目背景

随着高校招生规模不断扩大，宿舍管理工作日益复杂。传统的人工管理方式存在效率低下、数据更新不及时、统计困难等问题，难以满足现代化管理需求。为此，开发一套宿舍管理系统，通过信息化手段提升宿舍管理效率和服务水平。

### 1.3 项目目标

建设一个功能完善、操作便捷、安全可靠的宿舍管理平台，实现楼宇、宿舍、学生、辅导员等基础信息的统一管理，支持住宿申请、分配、调整、退宿等全流程管理，提供人脸识别等智能化功能，为管理人员提供数据分析和决策支持。

### 1.4 术语定义

- 楼宇：指学生宿舍楼，包含多间宿舍房间
- 宿舍：指楼宇中的具体房间，有固定的床位数量
- 辅导员：指负责宿舍管理的工作人员，可管理多栋楼宇
- 入住：指学生分配宿舍并入住的行为
- 退宿：指学生搬离宿舍的行为
- 调宿：指学生在不同宿舍之间调整的行为

## 2. 总体描述

### 2.1 系统定位

宿舍管理系统是一款面向高校宿舍管理场景的综合性信息化解决方案，采用 B/S 架构设计，用户无需安装客户端软件，通过浏览器即可访问使用。

### 2.2 用户群体

系统主要服务于三类用户：

系统管理员：负责系统的整体运维和基础数据管理，具有最高权限。

宿舍辅导员：负责所管辖楼宇的宿舍分配和日常管理工作，具有楼宇管理权限。

学生：查看个人住宿信息、申请调宿等，具有个人信息查看权限。

### 2.3 运行环境

服务器端：CentOS 7.6+ 或 Ubuntu 20.04+，JDK 1.8.0_200+，MySQL 8.0+，内存≥4GB，硬盘≥50GB。

客户端：支持主流浏览器（Chrome、Firefox、Edge 等），需要摄像头用于人脸识别。

## 3. 功能需求

### 3.1 用户管理模块

#### 3.1.1 用户登录

系统应支持账号密码登录和人脸识别登录两种方式。账号密码登录时，系统验证用户名和密码的正确性，验证通过则登录成功。人脸识别登录时，系统调用摄像头采集人脸图像，与已注册的人脸信息进行比对，匹配成功则登录成功。

#### 3.1.2 用户注册

系统支持管理员手动添加用户，也支持批量导入用户。添加用户时需要填写必要信息，如学号或工号、姓名、性别等，系统自动校验信息的唯一性。

#### 3.1.3 密码管理

用户登录后可修改个人密码，修改时需要输入原密码和新密码。管理员可重置任何用户的密码，重置后的密码为默认密码。

#### 3.1.4 人脸注册

用户可在个人中心进行人脸注册，系统调用摄像头采集人脸图像，提取人脸特征并保存。注册成功后可用于人脸登录。

### 3.2 楼宇管理模块

#### 3.2.1 楼宇列表查看

系统应以列表形式展示所有楼宇信息，包括楼宇名称、位置、辅导员等信息。支持按名称搜索。

#### 3.2.2 楼宇添加

管理员可添加新楼宇，填写楼宇名称、位置等信息。楼宇名称必须唯一，系统自动校验。

#### 3.2.3 楼宇编辑

管理员可编辑楼宇信息，修改名称、位置等。修改名称时需确保不与现有楼宇重名。

#### 3.2.4 楼宇删除

管理员可删除楼宇，删除前系统检查楼宇下是否有宿舍，如有宿舍则不允许删除。

#### 3.2.5 辅导员分配

系统支持为楼宇分配辅导员，一个楼宇可以分配给多个辅导员，一个辅导员也可以管理多栋楼宇。

### 3.3 宿舍管理模块

#### 3.3.1 宿舍列表查看

系统展示宿舍列表，可选择楼宇和楼层进行筛选。列表显示宿舍编号、楼层、入住人数、最大人数等信息。

#### 3.3.2 宿舍添加

管理员可添加新宿舍，选择所属楼宇，填写宿舍编号、楼层、最大人数等信息。宿舍编号必须唯一。

#### 3.3.3 宿舍编辑

管理员可编辑宿舍信息，修改楼层、最大人数等。宿舍编号一般不允许修改。

#### 3.3.4 宿舍删除

管理员可删除宿舍，删除前系统检查宿舍是否有学生入住，如有学生则不允许删除。

#### 3.3.5 入住统计

系统自动统计每个宿舍的已住人数和剩余床位，实时更新显示。

### 3.4 学生管理模块

#### 3.4.1 学生列表查看

系统展示学生列表，可按学号、姓名等条件搜索。列表显示学生基本信息和当前住宿状态。

#### 3.4.2 学生添加

管理员可添加新学生，填写学号、姓名、性别等信息。学号必须唯一。

#### 3.4.3 批量导入

系统支持从 Excel 文件批量导入学生信息，导入时自动校验数据格式和唯一性。

#### 3.4.4 学生编辑

管理员可编辑学生信息，修改姓名、性别等。学号一般不允许修改。

#### 3.4.5 学生删除

管理员可删除学生，删除前检查学生是否已入住，如已入住需先办理退宿。

### 3.5 辅导员管理模块

#### 3.5.1 辅导员列表查看

系统展示辅导员列表，显示基本信息和所管理的楼宇。

#### 3.5.2 辅导员添加

管理员可添加新辅导员，填写工号、姓名、性别等信息。工号必须唯一。

#### 3.5.3 楼宇分配

管理员可为辅导员分配管理的楼宇，支持多楼宇分配。

#### 3.5.4 辅导员编辑

管理员可编辑辅导员信息，修改姓名、性别等，可调整所管理的楼宇。

#### 3.5.5 辅导员删除

管理员可删除辅导员，删除前需解除与楼宇的关联关系。

### 3.6 住宿管理模块

#### 3.6.1 办理入住

管理员可为学生办理入住，选择学生和空闲宿舍，系统记录入住时间，更新宿舍入住人数。

#### 3.6.2 办理退宿

管理员可为学生办理退宿，系统更新宿舍入住人数，释放床位。

#### 3.6.3 调整宿舍

管理员可为已入住学生办理调宿，选择新宿舍后系统自动完成原宿舍退宿和新宿舍入住。

#### 3.6.4 住宿记录查询

系统记录学生的所有住宿历史，包括入住时间、退宿时间、宿舍变更等信息，支持查询。

### 3.7 数据导出模块

#### 3.7.1 学生信息导出

系统支持将学生信息导出为 Excel 文件，包含学号、姓名、性别、住宿状态等信息。

#### 3.7.2 宿舍信息导出

系统支持将宿舍信息导出为 Excel 文件，包含宿舍编号、楼宇、楼层、入住情况等信息。

#### 3.7.3 住宿统计导出

系统支持将住宿统计数据导出为 Excel 文件，包含各楼宇、各宿舍的入住率等信息。

## 4. 性能需求

### 4.1 响应时间

核心操作（如登录、查询、添加等）响应时间不超过 3 秒。非核心操作（如数据导出、批量导入等）响应时间不超过 5 秒。

### 4.2 并发性能

系统应支持至少 100 人同时在线操作，关键业务操作不出现明显延迟。

### 4.3 系统可用性

系统可用性达到 99% 以上，每月故障时间不超过 7.2 小时。

### 4.4 数据处理能力

系统应能处理至少 10000 名学生、1000 间宿舍、100 名辅导员的数据量。

## 5. 安全需求

### 5.1 用户认证

系统应实现用户身份认证，未登录用户不能访问系统功能。支持账号密码和人脸识别两种认证方式。

### 5.2 权限控制

系统应实现基于角色的权限控制，不同角色只能访问授权的功能和数据。

### 5.3 数据安全

用户密码等敏感数据应加密存储，防止数据泄露。

### 5.4 日志审计

系统应记录关键操作日志，包括操作人、操作时间、操作内容等，便于追溯。

## 6. 接口需求

### 6.1 外部接口

系统需集成百度 AI 人脸识别服务，调用其 API 进行人脸检测和识别。系统需集成阿里云 AI 助手服务，提供智能问答功能。

### 6.2 内部接口

系统各模块之间通过 RESTful API 进行数据交互，接口采用 JSON 格式。

## 7. 数据需求

### 7.1 数据准确性

系统应保证数据的准确性和一致性，关键数据应进行校验。

### 7.2 数据备份

系统应支持数据库定期备份，防止数据丢失。

### 7.3 数据恢复

系统应支持数据恢复功能，在数据损坏时能够恢复。
